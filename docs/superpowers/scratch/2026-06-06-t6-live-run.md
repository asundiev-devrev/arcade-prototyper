## Findings (real, for follow-up — not #3 blockers)
1. **`importComponentByKeyAsync` hangs (>30s) even for a same-file component.**
   The Slice 0 async-kick+poll did NOT complete here either (globalThis IIFE
   appears torn down between figma_execute calls). WORKAROUND that worked: when
   the active file IS the library, instantiate from the LOCAL component-set node
   (`getNodeByIdAsync(setNodeId)` → pick the variant child → `.createInstance()`).
   The real #3 bridge should prefer local-node instancing when in-file, and treat
   importByKey as the cross-file (separate output file) path with a longer budget
   / true background poll.
2. **`setText` node-targeting: `lowest-depth`/`[0]` is wrong for Bubble.** The
   Bubble component has multiple internal TEXT nodes (template sender/receiver/
   username variants); `findAllWithCriteria(TEXT)[0]` hit a hidden template node,
   so the visible message text did NOT update (bubbles show default placeholder
   copy). FIX: the #2 ChatBubble entry needs a precise `textNode` hint
   (`by-name` with the real message layer name, or the visible/largest text node),
   not the lowest-depth heuristic. Exactly the risk the #3 spec flagged.
3. **Per-call op budget:** 15 instance-creates + 15 font-load+setText exceeded a
   single figma_execute 30s window; ran in batches of 5. The real bridge executor
   should chunk ops, and load each distinct font once up front.

## Bottom line
The planner→executor contract, instance creation, variant mapping, and auto-layout
all work against the real Bridge. Two execution-layer refinements (local-node
instancing; precise text-node targeting per component) are needed for fidelity —
both are bridge/#2-mapping tweaks, not planner/executor-logic bugs.
