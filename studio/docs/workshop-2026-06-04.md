# Arcade Studio — Designer Workshop Run Sheet

**When:** Thursday 2026-06-04 · 60 min · online
**Audience:** designers (some have dabbled) · app pre-installed
**Must-have goal:** every designer can build a prototype solo by the end
**Ideal goal:** they leave *wanting* to use it

**Golden rule:** ~35 of 60 min is hands-on-keyboard. Talk less, let them build.

Everything below is validated live against **Studio 0.26.1**. Exact prompts are pre-tested — use them verbatim.

---

## The spine: one screen, progressively enriched

We build **one Computer chat screen** and keep layering on it. Each step adds a real capability — no topic-switching. The screen the room ends with is clickable, wired to real data, and carries a production blueprint.

| Beat | Who builds | What it adds |
|---|---|---|
| 1. Hook | facilitator | the core loop: prompt → frame |
| 2. Orient | — | the 3 moves |
| 3. Guided build | everyone | reliability + the iterate loop |
| 4. Real data ⭐ | everyone | live Computer sessions/chats |
| 5. Clickable | everyone | multi-frame flow (FrameLink) |
| 6. Production blueprint ⭐ | facilitator | prototype → real code (the finale) |
| (Prompting + recovery) | — | woven in before free-play |

## 0 · Pre-flight (day before — not live)
- Confirm everyone is on **0.26.1** (update banner, or re-send DMG). Pre-0.26.1 builds have the silent failures we fixed — this matters.
- Confirm DevRev PAT + AWS Bedrock connected.
- Homework: *"bring one screen from your current work you wish you could click through."*

## 1 · Hook (0:00–0:05) — facilitator, live
Cold-open build. Type one sentence, watch a real, interactive frame appear.

> **Prompt (verbatim):** `Build a Computer chat screen with the artefacts panel open`

- ✅ Verified: builds `<ComputerScene withCanvasPanel />` in ~20s — full scene, sessions/chats sidebar, populated transcript, artefacts panel.
- ⚠️ **Must name a variant** ("with the artefacts panel open"). A bare *"build a Computer chat screen"* hits the dedup guard → it says "that already exists" and builds nothing. (This is a Part-2 teaching point.)

## 2 · Orient (0:05–0:12) — no build
Use the frame from step 1. Point at: the prompt box → "Working…" → the frame lands in the grid. Name the **3 moves** they'll use all session: **prompt · iterate · (later) share/lift.** Hide Settings/Bedrock/packaging.

## 3 · Guided build, lockstep (0:12–0:28) — everyone
Everyone types the same prompt; wait for the room. Then **one iteration** — that's the real lesson: prototyping is a conversation.

> **Build (verbatim):** `Build a settings page for notification preferences — a few toggle rows in a card`
> **Iterate (verbatim):** `split the notification settings into two columns`

- ✅ Verified: settings build ~55s; the split applies cleanly. Settings is the most reliable kit output — safe for 20 people at once.
- Dabblers float as helpers.
- **Checkpoint (must-have goal):** by 0:28 everyone has built + iterated one frame.

## 4 · Real data ⭐ (0:28–0:40) — everyone tries live
The differentiator nobody else has: pipe **their own** Computer conversations into the prototype.

> **Prompt (verbatim):** `use my real Computer sessions and chats instead of mock data`

- ✅ Verified: fetches live threads, splits **Sessions** (your Computer threads) from **Chats** (human DMs), loads real transcripts.
- 🚫 **Do NOT prefix with `@Computer`.** `@Computer` routes to the conversational agent → text reply, **builds no frame**. The data-into-prototype trigger is the plain sentence above. (`@Computer` is a separate feature — out of scope here.)
- **Beta honesty:** it's new and can be rough. If a result looks messy, that's the live prompting lesson (see Part 2), not a failure.
- **Guardrails for 20-at-once:** helpers on hand; a known-good re-prompt ready; some designers may have few Computer sessions → that's fine, the structure still shows.

## 5 · Clickable (0:40–0:48) — everyone
Make it a real click-through, not a static mock.

> **Prompt (verbatim):** `add an empty-state version of this screen as a second frame, and when I click New Chat take me to it`

- ✅ Verified: creates a 2nd frame + wires the New Chat button with FrameLink → navigates between frames in the viewport.

## 6 · Production blueprint ⭐ (0:48–0:56) — facilitator, the finale
**The closer.** Reframes the whole tool: prototypes aren't throwaways.

- Show the `LIFT.xml` that rides next to every frame the room built.
- Walk 1–2 mappings out loud: *"it already knows your Switch → the production Switch, Bell → ICON_TYPES.NOTIFICATION, this token → this Tailwind class."*
- The pitch: *"Hand this to your engineer's Claude Code and it follows the recipe to turn your prototype into real devrev-web code — minutes, not a rebuild. Your design doesn't die in a mock."*

- ✅ Verified end-to-end against real devrev-web: ~80%-mergeable production code, minor cleanup. **7/10 — real, not vapor.**
- **Honesty bar:** say *"~80% there, your engineer does minor cleanup"* — NOT "one-click / auto-merges." Over-claiming invites the "does it really work?" pushback.

## 7 · Free play + close (0:56–60:00)
- Designers build their homework screen (helpers roam) — *this is where excitement is manufactured: their project, working.*
- Close: help channel · one concrete next action (*"ship one prototype into a real review this week"*) · where to find the run sheet.

## Part 2 — Prompting well + recovering (woven in before free-play, ~6 min)

Frame positively: *"It's a conversation. Here's how to steer it — most problems are preventable."*

### Half A — Prompt well (prevents most failures)

| Principle | ✗ Weak | ✓ Strong |
|---|---|---|
| One screen at a time | "build chat, settings, and a dashboard" | "build the chat screen first" |
| Be specific or get the default | "build a Computer screen" → *"already exists"* | "…**with the artefacts panel**" (name a variant) |
| Describe the end state | "use a flex sidebar layout" | "chat list on the left, conversation on the right" |
| Point at the thing | "fix the button" | "the blue button, top-right — make it say Save" |
| Name real data plainly | "add my data" | "use my real Computer sessions and chats" (no `@`) |
| Iterate, don't rewrite | re-describe the whole screen | "now just change the header title" |

**Golden rule to drill:** *small steps beat big rewrites.* Almost every failure is one giant prompt.

### Half B — Recover when it drifts

| What you see | What it means | The move |
|---|---|---|
| "That already exists," builds nothing | Prompt was generic | Add a specific variant ("…empty state", "…with the panel") |
| "Working…" but slow | Big frame writes take ~1–2 min | Wait. It's working. >3 min frozen → Stop + re-prompt smaller |
| Blank / broken frame | It hallucinated | Don't pile fixes on a broken base — "go back to how it looked, then instead do Y" |
| Ignored your change | You typed while it was still working, or too vague | **Wait for it to fully finish**, then re-prompt — point at the element |
| Real-data looks messy | Beta roughness | "keep my sessions and chats in separate groups" — be explicit about structure |

Every recovery move = a Half-A principle applied after the fact: **ask for less, be more specific.**

## Arcade Studio vs Cursor — when to use which

A question that *will* come up. Answer it proactively and honestly — they solve different problems, and over-claiming hurts trust.

**One-line framing:** *Cursor is an engineer's AI code editor for real codebases. Arcade Studio is a designer's prototyping tool that speaks DevRev's design language and hands work off to engineering.* They're complementary, not competitors.

| | **Arcade Studio** | **Cursor** |
|---|---|---|
| Built for | Designers (no code required) | Engineers (you write/read code) |
| You work in | Plain sentences → frames appear | A real codebase + an AI editor |
| Design system | DevRev's baked in (Arcade kit, tokens, Computer composites) | None — it knows your repo, not our design language |
| Setup | Open the app, type | Clone repo, install deps, learn the codebase |
| DevRev data | Pulls your real Computer sessions/chats live | Only what's in the repo you opened |
| Output | A clickable prototype + a production blueprint (LIFT) | Real committed code in a real project |
| Speed to first screen | Seconds | Depends on the codebase |

**Use Arcade Studio when:**
- You're exploring an idea and want to *see and click* it fast.
- You want DevRev-accurate UI without hand-building chrome.
- You want to show a flow to a PM/eng and hand off something they can actually build from.
- You don't write production code (or don't want to, for a throwaway exploration).

**Use Cursor when:**
- You're an engineer making real changes in a real codebase.
- You need to ship merged code, run tests, touch backend/logic, not just UI.
- The work isn't a prototype — it's the product.

**The honest seam (say this out loud):** Arcade Studio doesn't *replace* the engineering step — its **LIFT blueprint is the bridge to it.** You prototype in Studio; your engineer takes the frame + blueprint into their editor (Cursor, Claude Code, whatever) and turns it into ~80%-there real code with minor cleanup. Studio is the front of the pipe, not the whole pipe.

**Don't say:** "Studio replaces Cursor" or "you'll never need to code again." Designers will immediately test that claim and it'll break. The credible, exciting message is *"your prototype now carries enough structure that it survives the handoff to engineering"* — that's the genuinely new thing.

## Features NOT demoed (and why)
- **`@Computer` mention** — separate feature (ask the agent a question); doesn't build frames. Out of the core arc.
- **Multiplayer / shared commenting** — unfinished (6 interlocking bugs). Tease with a screenshot if asked; don't demo live.
- **Share-to-web link** — works, but the production-blueprint finale is the stronger sharing story, so it's demoted.

## Risk notes
- Live generation is non-deterministic + online. The homework + helpers + verbatim pre-tested prompts are the insurance.
- If step 4 (real data) wobbles for someone, route it to the Part-2 lesson, not a panic.
- Pre-built fallback: have the step-1 hook frame already made on your machine in case live generation stalls at the open.
